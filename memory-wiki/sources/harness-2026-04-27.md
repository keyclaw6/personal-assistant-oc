# Agent Harness Research

## L0

Agent memory should expose a small semantic facade. The model should see `mem search`, `mem get`, `mem put`, and `mem check`, not the full maintenance script graph.

## L1

Benchmark-oriented agent work points to harness design as a major performance lever: prompt structure, context selection, tool names, tool descriptions, and traces can change outcomes without changing the base model. For this memory system, the ergonomic answer is a compact facade over the existing file-based machinery.

## L2

Observed principles:

- Keep the visible tool surface small enough for the model to choose reliably.
- Use names that are short but still semantic. Opaque abbreviations reduce token count but remove meaning.
- Separate recall tools from maintenance tools.
- Prefer progressive disclosure: index, search, get one item, then stop.
- Keep outputs compact by default, with JSON available for scripted runtimes.

Applied here:

- `npm run mem -- search "query"` wraps deterministic search.
- `npm run mem -- get <id-or-path>` fetches one selected page or claim.
- `npm run mem -- put ...` captures raw unreviewed memory.
- `npm run mem -- check` runs the memory health gate.

Maintenance commands such as compile, report, eval, and smoke remain available but should not be the normal model-facing interface.
